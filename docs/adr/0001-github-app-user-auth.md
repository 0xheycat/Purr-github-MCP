# ADR 0001: GitHub App user authentication without replacing Purr tools

Status: accepted

Date: 2026-07-16

Base ref: `feat/oauth-chatgpt-durable` at `4cb09ef3eef5988931270df56f7f3f8a0d2cb1b6`

## Decision

Add GitHub user authentication as an adapter inside the existing durable OAuth wrapper. Keep the current MCP server, its 35-tool registry, handlers, input schemas, annotations, transport, and safety policy unchanged.

The public flow becomes:

```text
ChatGPT OAuth client
  -> Purr OAuth authorization request
  -> GitHub App web authorization
  -> encrypted GitHub user credential
  -> Purr MCP access/refresh grant containing only a credential reference
  -> credential resolver
  -> unchanged Purr MCP upstream
  -> GitHub API using the authenticated user's token
```

The existing owner path remains available:

```text
SERVER_TOKEN -> existing GITHUB_TOKEN -> unchanged full legacy access
```

## Maintained upstream references

Use these projects as implementation and test references rather than designing provider OAuth from zero:

- `github/github-mcp-server`
  - audited main tree: `8ac674b0562bf2d2cccaea2965d32fcb2511cdbe`
  - `docs/oauth-login.md`
  - `internal/oauth/flow.go`
  - `internal/oauth/manager.go`
  - `internal/oauth/callback.go`
  - `pkg/http/oauth/oauth.go`
  - `pkg/http/middleware/token.go`
  - `pkg/http/middleware/scope_challenge.go`
- `octokit/oauth-app.js`
  - GitHub App web-flow URL, code exchange, token inspection, refresh, scoped tokens, and lifecycle events
- `octokit/webhooks.js`
  - webhook signature verification and typed lifecycle events
- `octokit/auth-app.js`
  - installation authentication only when a tool later requires an installation token

Do not fork the official Go server and do not port its GitHub tool registry. Its OAuth state machine, fail-closed behavior, and tests are references for the Node adapter.

## Dependency boundary

Production dependencies are limited to official Octokit packages required by the adapter:

```text
@octokit/oauth-app
@octokit/webhooks
```

`@octokit/auth-app` is added only if installation-token execution is required by a verified endpoint. The current target is GitHub App user-to-server authentication, because every existing Purr tool already accepts `ctx.githubToken`.

Use the package lock generated by `npm install`; do not hand-copy dependency internals.

## Invariants

1. `src/server.js` tool definitions and handlers are not rewritten.
2. `src/extensions.js` tool definitions and handlers are not rewritten.
3. Tool names, schemas, annotations, and count remain identical.
4. GitHub tokens never appear in MCP access tokens, browser responses, logs, or tool output.
5. OAuth state is one-time, expiration-bound, and linked to one Purr authorization request.
6. The callback resolves the GitHub account with `GET /user` before issuing the MCP authorization code.
7. The MCP grant stores `githubCredentialRef` and a stable subject such as `github:<numeric-id>`.
8. A request resolves one fresh GitHub token before forwarding to the unchanged upstream.
9. Purr MCP scopes, GitHub permissions, and existing safety policy must all allow a mutation.
10. Direct valid `SERVER_TOKEN` access continues using the existing server-side `GITHUB_TOKEN`.
11. GitHub authorization revocation marks the credential unusable and prevents future refresh.
12. No phase advances while its gate is failing.

## Phase lock

### Phase 1: reference and architecture

Gate:

- maintained references locked
- dependency boundary locked
- all 35 tools mapped
- no runtime tool change

### Phase 2: GitHub OAuth identity

Gate:

- GitHub login and callback pass
- callback replay rejected
- provider token encrypted at rest
- resolved numeric GitHub identity matches the login
- two users remain isolated

### Phase 3: credential routing and parity

Gate:

- 35 tools before equals 35 tools after
- `get_authenticated_user` returns the connected account
- read and write fixtures use the connected user's token
- legacy owner route remains operational
- protected-branch and secret guards still reject unsafe writes

### Phase 4: lifecycle and production hardening

Gate:

- refresh rotation passes
- revocation and installation lifecycle pass
- ChatGPT browser end-to-end flow passes
- live tools/list and tools/call smoke pass
- rollback instructions validated

There is no Phase 5.

## Rejected alternatives

### Replace Purr with GitHub's official MCP server

Rejected because it changes tool names, schemas, behavior, and safety policy instead of adding identity to the working Purr server.

### Build provider OAuth and refresh logic from scratch

Rejected because maintained Octokit packages already implement the provider-specific protocol and lifecycle.

### Continue proxying every OAuth user through one shared `GITHUB_TOKEN`

Rejected because users would share one GitHub identity and repository authority.

### Remove tools when a GitHub user lacks permission

Rejected for the full-access ChatGPT installation target. The 35-tool catalog remains stable; GitHub and Purr deny unauthorized execution with structured errors.
